# Агентский навык: 1C Repo Map

Этот файл описывает, как агенты (ИИ-ассистенты) должны работать с репозиторными картами 1С.

## Когда использовать

Используй этот навык, когда:
- Пользователь работает с конфигурацией 1С (файлы `.bsl`, `.xml`, структура `Catalogs/`, `Documents/`)
- Нужно найти объект метаданных, модуль, форму или регистр
- Пользователь просит исправить ошибку, добавить реквизит, доработать отчет
- В рабочей директории есть папка `repo_map/` с сгенерированными картами

## Принцип работы

**Всегда читай карты ДО чтения исходных файлов.**

Карты сжимают объем контекста на 80-90%, давая ИИ навигацию без необходимости читать тысячи XML-файлов.

## Алгоритм действий

### Шаг 1: Проверить наличие карт

```
Если существует папка repo_map/:
    → Читай README.md
    → Переходи к шагу 2
Иначе:
    → Предложи пользователю сгенерировать карты:
      "Конфигурация большая. Сгенерировать репозиторные карты для навигации?"
    → Если пользователь согласен — запусти generate_repo_map.py
```

### Шаг 2: Определить тип задачи

| Тип задачи | Какие карты читать |
|------------|-------------------|
| Работа со справочником | `README.md` → `Catalogs.md` → `subsystems.md` |
| Работа с документом | `README.md` → `Documents.md` → `subsystems.md` |
| Работа с регистром | `README.md` → `InformationRegisters.md` или `AccumulationRegisters.md` |
| Работа с отчетом/обработкой | `README.md` → `Reports.md` или `DataProcessors.md` |
| Работа с общим модулем | `README.md` → `CommonModules.md` → `modules_graph.md` |
| Не знаю, где искать | `README.md` → `objects_index.md` или `keywords.md` |

### Шаг 3: Найти объект

1. Ищи объект по имени в соответствующей карте типа
2. Если не нашел — ищи в `objects_index.md` (алфавитный индекс)
3. Если не нашел — ищи в `keywords.md` (по ключевым словам)
4. Если не нашел — используй grep по именам файлов (fallback)

### Шаг 4: Изучить структуру

В карте объекта найди:
- **Реквизиты/Измерения/Ресурсы** — что может понадобиться для задачи
- **Формы** — какие формы есть (ФормаЭлемента, ФормаСписка и т.д.)
- **Модули** — ObjectModule.bsl, ManagerModule.bsl, Form/Module.bsl
- **Подсистемы** — к какой функциональной области относится

### Шаг 5: Прочитать только нужные файлы

Не читай всю папку объекта! Прочитай только:
1. **Модуль объекта** (`Ext/ObjectModule.bsl`) — логика проведения, события
2. **Модуль менеджера** (`Ext/ManagerModule.bsl`) — программный интерфейс
3. **Модуль формы** (`Forms/*/Ext/Form/Module.bsl`) — если задача про форму
4. **Описание формы** (`Forms/*/Ext/Form.xml`) — если нужны элементы формы

**Максимум 3-5 файлов на задачу.**

### Шаг 6: Анализ зависимостей (при необходимости)

Если нужно понять, кто вызывает модуль или кого вызывает:
- Читай `modules_graph.md`
- Ищи имя модуля в графе зависимостей
- Переходи к чтению связанных модулей (макс. 2-3)

## Примеры промптов для пользователя

### Пример 1: Найти и исправить

```markdown
Пользователь: "Исправь проведение документа Прием"

Действия агента:
1. Читает README.md (понимает масштаб)
2. Читает Documents.md, находит раздел ## Прием
3. Видит: модули ObjectModule.bsl, ManagerModule.bsl
4. Видит: связанные регистры из subsystems.md
5. Читает Documents/Прием/Ext/ObjectModule.bsl
6. Читает Documents/Прием/Ext/ManagerModule.bsl
7. Дает ответ с конкретным исправлением
```

### Пример 2: Добавить реквизит

```markdown
Пользователь: "Добавь поле 'Комментарий' в справочник Пациенты"

Действия агента:
1. Читает Catalogs.md, находит ## Пациенты
2. Видит текущие реквизиты и формы
3. Видит, что есть ФормаЭлемента
4. Читает Forms/ФормаЭлемента/Ext/Form.xml (структура формы)
5. Читает Forms/ФормаЭлемента/Ext/Form/Module.bsl
6. Дает инструкцию: куда добавить реквизит и элемент формы
```

### Пример 3: Не знаем, где искать

```markdown
Пользователь: "Где настраивается интеграция с DICOM?"

Действия агента:
1. Читает keywords.md, ищет "DICOM"
2. Находит: CommonModules/DICOM, DataProcessors/DICOMWorkList
3. Читает CommonModules.md → раздел DICOM
4. Читает modules_graph.md — кто вызывает DICOM
5. Дает навигацию по всем DICOM-объектам
```

## Ограничения

1. **Карты не заменяют чтение кода** — они только направляют к правильным файлам
2. **Размер карт** — objects_index.md может быть >1 МБ, читай его выборочно (grep по ключевым словам)
3. **Актуальность** — карты могут устареть; если объект не найден, предложи перегенерировать
4. **Модули** — граф зависимостей строится только для 500 модулей (параметр --max-modules)

## Генерация карт

Если карт нет или они устарели:

```bash
python src/generate_repo_map.py \
  --config "/путь/к/конфигурации" \
  --output "./repo_map" \
  --max-modules 500
```

Вывод скрипта:
- Список сгенерированных `.md` файлов
- Общий размер
- Количество проанализированных объектов

## Структура выходных карт

```
repo_map/
├── README.md                 # Точка входа, статистика
├── objects_index.md          # Алфавитный индекс
├── subsystems.md             # Иерархия подсистем
├── modules_graph.md          # Зависимости модулей
├── keywords.md               # Поисковый индекс
├── Catalogs.md               # Справочники
├── Documents.md              # Документы
├── InformationRegisters.md   # Регистры сведений
├── AccumulationRegisters.md  # Регистры накопления
├── CommonModules.md          # Общие модули
├── DataProcessors.md         # Обработки
└── Reports.md                # Отчеты
```

## Правила именования

- Не сокращай имена объектов 1С — используй точные имена из конфигурации
- При ссылке на файл используй относительный путь от корня конфигурации: `Documents/Прием/Ext/ObjectModule.bsl`
- При ссылке на карту используй markdown-ссылку: `[Documents.md](Documents.md)`
